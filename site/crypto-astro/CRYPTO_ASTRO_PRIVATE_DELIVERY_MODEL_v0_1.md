# Crypto-Astro Private Delivery Model v0.1

status: DELIVERY_MODEL_LOCKED
scope: private research delivery / asset-context foundation
repo: AiBhrigu/phi-cosmography-open
target: Crypto-Astro as first proof corridor for a broader asset-context research system

## Purpose

This artifact defines the private delivery model for Crypto-Astro.

Crypto-Astro is the first proof corridor. The delivery architecture must remain reusable for other asset classes and research contexts.

The public Crypto-Astro surface remains a proof/context/boundary layer. The value layer is delivered through controlled private research notes, not through public feeds, public APIs, public JSON, or public subscriber report archives.

## Scope

This model applies to:

* Crypto-Astro public surface
* private Crypto-Astro research delivery
* future asset-context research delivery
* protected-output governance
* operator-reviewed synthesis flow

This model does not authorize runtime, backend/API, payment automation, formula execution, A/E numeric activation, TVL implementation, trading signals, forecasts, price targets, or investment recommendations.

## Current public layer

The public layer may show:

* public-safe market context
* low-resolution status
* proof state
* governance and closure artifacts
* boundary notes
* delayed or synthetic demo examples
* static schema without active weights
* no-trading-signal / no-forecast copy

The public layer is not the value delivery layer.

## Private delivery layer

The private delivery layer may include:

* request-based research notes
* asset-specific context
* temporal / event-window context
* evidence and source notes
* market context
* pattern synthesis
* uncertainty and risk boundary
* operator-reviewed interpretation
* controlled snapshot interpretation

Private delivery must preserve safety boundaries and must not present outputs as financial advice.

## Request intake

A request should capture:

* asset or assets
* date or time window
* user context or research question
* market situation
* what the user wants to understand
* explicit boundary agreement

Accepted request framing:

* market-cycle context
* temporal context
* asset-specific research
* scenario framing
* what to watch
* uncertainty review

Rejected or reframed request framing:

* buy / sell / hold
* entry / exit
* leverage
* price target
* guaranteed outcome
* urgent trade instruction
* investment recommendation

## Gate / refusal rules

Before delivery, every request must pass a boundary gate.

Reject or reframe requests that ask for:

* trading signals
* price targets
* investment advice
* guaranteed scenarios
* urgent market instructions
* public reusable alpha tables
* formula weights
* hidden thresholds
* validation datasets
* private client context disclosure

## Delivery packet format

The first delivery product is:

Crypto-Astro Private Research Note v0.1

Recommended structure:

1. Request summary
2. Asset context
3. Temporal / event-window context, including C/T Temporal · Ephemerides Support when available
4. Market context
5. Pattern synthesis
6. What to watch
7. What is not concluded
8. Risk / uncertainty boundary
9. Source / proof note
10. No financial advice seal

This is a private research note, not a trading signal.

C/T Temporal context may be included as a bounded research-note block for observation-window framing, cycle context, ephemerides support state, and provenance. It must remain context-only and must not produce forecasts, trading signals, price targets, or investment advice.

## Archive / proof trail

Each private delivery should have:

* request_id
* delivery_id
* created_at_utc
* operator_review_status
* boundary_status
* local report path
* sha256
* memory capsule

The archive trail protects continuity, reduces drift, and keeps delivery history auditable.

## Asset-agnostic expansion rule

Crypto is the first proof domain.

The delivery model must not be hard-coded as crypto-only.

Future asset-context corridors may include:

* crypto assets
* equities
* commodities
* indices
* macro-sensitive assets
* other research contexts

Any future asset class must preserve the same boundary:

public = proof / context / trust
private = interpretation / synthesis / delivery
protected = formulas / validation / ORION core

## Protected core boundary

The following must remain protected:

* ORION reasoning
* formula weights
* validation datasets
* hidden thresholds
* runtime-generated alpha
* private client context
* paid/manual synthesis logic
* future protected calculation core

## Governance

Public HTML changes require a scoped PR route.

Delivery model changes require PR review.

Formula execution, A/E numeric activation, runtime/backend/API, live adapters, TVL implementation, payment route, and subscriber-grade delivery each require separate scoped plans.

No public output should become a copyable feed of subscriber-grade value.

* live adapters
* payment route
